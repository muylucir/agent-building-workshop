# Spec Compliance Evaluation

구현된 에이전트 코드가 PATH 명세서를 얼마나 충실하게 구현했는지 평가하는 기준.

## 평가 방법

1. 명세서 파일과 구현된 코드 디렉토리를 모두 읽는다
2. 아래 7개 평가 영역별로 체크리스트를 수행한다
3. 각 영역별 점수(0~100)와 누락/불일치 항목을 리포트한다
4. 종합 점수와 심각도별 이슈 목록을 출력한다

## 평가 영역

### 1. 에이전트 완전성 (Agent Completeness)

명세서에 정의된 모든 에이전트/스테이지가 코드에 존재하는가.

| 체크 항목 | 방법 |
|----------|------|
| 명세서 Agent Components 테이블의 모든 에이전트가 코드에 클래스로 존재하는가 | 명세서 테이블 행 수 vs 코드 클래스 수 비교 |
| 에이전트 이름이 명세서와 대응되는가 | 명세서 Agent Name → 코드 클래스명 매핑 확인 |
| 각 에이전트의 책임(Role)이 코드에 반영되어 있는가 | 클래스 docstring 또는 구현 로직 확인 |

점수 산출: `(구현된 에이전트 수 / 명세서 에이전트 수) × 100`

### 2. 프롬프트 충실도 (Prompt Fidelity)

명세서의 시스템 프롬프트가 코드에 그대로 사용되었는가.

| 체크 항목 | 방법 |
|----------|------|
| 명세서의 핵심 XML 태그가 코드 프롬프트에 존재하는가 | `grep`으로 코드 내 `<role>`, `<context>`, `<instructions>`, `<constraints>`, `<output_format>` 태그 검색 |
| 명세서 프롬프트의 첫 줄과 마지막 줄이 코드에 동일하게 존재하는가 | 명세서에서 각 프롬프트의 첫 줄/마지막 줄을 추출 → `grep`으로 코드에서 검색 |
| 명세서 프롬프트의 고유 문구(도메인 용어, 규칙 번호 등)가 코드에 존재하는가 | 명세서 프롬프트에서 5개 이상의 고유 문구를 샘플링 → `grep`으로 코드에서 각각 검색. 모두 발견되면 verbatim으로 판정 |
| 프롬프트 길이가 명세서와 유사한가 | 명세서 프롬프트 줄 수와 코드 프롬프트 줄 수를 비교. ±10% 이내면 통과, 50% 이상 차이나면 요약/재작성으로 판정 |
| Example User Prompt 구조가 템플릿 함수로 변환되었는가 | `get_*_prompt()` 함수 존재 및 구조 확인 |

점수 산출: 각 에이전트별로 위 5개 체크 항목을 수행. 고유 문구 검색에서 누락이 발견되거나 길이가 50% 이상 차이나면 해당 에이전트 0점.

### 3. 도구 완전성 (Tool Completeness)

명세서에 정의된 모든 도구가 코드에 구현되었는가.

| 체크 항목 | 방법 |
|----------|------|
| 명세서 Tool Definitions의 모든 도구가 `@tool` 함수로 존재하는가 | 명세서 도구 이름 목록 vs 코드 `@tool` 함수 이름 목록 비교 |
| 도구 함수의 파라미터가 명세서 Signature와 일치하는가 | 파라미터 이름·타입·기본값 비교 |
| 도구 함수의 docstring이 명세서 Purpose를 반영하는가 | docstring 존재 및 내용 확인 |
| 각 에이전트에 명세서에서 지정한 도구가 연결되어 있는가 | 에이전트 `tools=[...]` 목록과 명세서 도구 목록 비교 |

점수 산출: `(구현된 도구 수 / 명세서 도구 수) × 100`, 파라미터 불일치 시 해당 도구 -50%

### 4. 상태 관리 완전성 (State Management)

명세서의 Shared/Session/Persistent State가 코드에 반영되었는가.

| 체크 항목 | 방법 |
|----------|------|
| Shared State 테이블의 모든 필드가 코드에 정의되어 있는가 | 명세서 필드 목록 vs Pydantic 모델 또는 dict 키 비교 |
| Session State 필드가 세션 관리 코드에 존재하는가 | 에이전트 인스턴스 속성 또는 세션 딕셔너리 확인 |
| Persistent State 필드에 대한 저장/조회 로직이 존재하는가 | mock 저장소 또는 파일 I/O 코드 확인 |
| 상태 필드의 작성 주체/소비 주체가 명세서와 일치하는가 | 코드에서 해당 필드를 쓰는/읽는 에이전트 추적 |

점수 산출: `(구현된 상태 필드 수 / 명세서 상태 필드 수) × 100`

### 5. 에러 처리 완전성 (Error Handling)

명세서의 Error Handling 테이블이 코드에 반영되었는가.

| 체크 항목 | 방법 |
|----------|------|
| 명세서 오류 시나리오별로 try/except 또는 검증 로직이 존재하는가 | 명세서 오류 시나리오 목록 vs 코드 예외 처리 블록 비교 |
| 복구 전략(재시도, 지수 백오프 등)이 명세서 명시대로 구현되었는가 | 재시도 횟수, 백오프 간격 등 파라미터 확인 |
| 최종 Fallback 경로가 구현되어 있는가 | 재시도 실패 후 대체 로직 존재 확인 |

점수 산출: `(구현된 오류 시나리오 수 / 명세서 오류 시나리오 수) × 100`

### 6. Mock 품질 (Mock Quality)

도구의 mock 구현이 현실적이고 명세서 예시와 일관되는가.

| 체크 항목 | 방법 |
|----------|------|
| mock이 단순 더미(`return "mock"`)가 아닌 입력 기반 로직을 포함하는가 | mock 함수 코드 복잡도 확인 |
| mock 반환값이 명세서 `<output_format>` JSON 구조와 일치하는가 | 반환 타입/필드 비교 |
| 명세서 `<examples>` 블록의 입력을 mock에 넣으면 예시와 유사한 출력이 나오는가 | 예시 입력으로 mock 함수 호출 시뮬레이션 |
| 에러 시뮬레이션이 지원되는가 | 특수 입력값 처리 코드 존재 확인 |
| `tools/`와 `mocks/`가 분리되어 있는가 | 디렉토리 구조 확인 |

점수 산출: 체크 항목별 20점씩 배분

### 7. 아키텍처 일치도 (Architecture Alignment)

구현된 코드의 전체 구조가 명세서의 설계 패턴과 일치하는가.

| 체크 항목 | 방법 |
|----------|------|
| Automation Level에 맞는 프로젝트 구조인가 | Agentic AI → `agents/`, AI-Assisted Workflow → `stages/` + `pipeline.py` |
| Layer3 패턴이 올바르게 구현되었는가 | 명세서 Layer3 → Strands 멀티에이전트 패턴 매핑 확인 |
| 에이전트 간 호출 흐름이 명세서 Sequence Diagram과 일치하는가 | 코드의 호출 순서 vs 명세서 다이어그램 비교 |
| 병렬 실행이 명세서에서 병렬로 정의된 부분과 일치하는가 | `asyncio.gather()` 등 병렬 코드 vs 명세서 Fan-out/Fan-in 비교 |

점수 산출: 체크 항목별 25점씩 배분

## 리포트 형식

```markdown
# 명세서 충실도 평가 리포트

## 요약
- 명세서: <파일명>
- 구현 디렉토리: <경로>
- 종합 점수: <0~100> / 100
- 등급: S (95+) / A (85+) / B (70+) / C (55+) / D (55 미만)

## 영역별 점수

| 영역 | 점수 | 비고 |
|------|------|------|
| 에이전트 완전성 | /100 | |
| 프롬프트 충실도 | /100 | |
| 도구 완전성 | /100 | |
| 상태 관리 완전성 | /100 | |
| 에러 처리 완전성 | /100 | |
| Mock 품질 | /100 | |
| 아키텍처 일치도 | /100 | |

## 이슈 목록

### 🔴 Critical (누락)
- [ ] <명세서에 있으나 코드에 없는 항목>

### 🟡 Warning (불일치)
- [ ] <명세서와 다르게 구현된 항목>

### 🔵 Info (개선 권장)
- [ ] <동작하지만 개선 가능한 항목>
```
