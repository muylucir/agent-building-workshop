# Mock 구현 패턴

도메인에 무관하게, 도구의 **성격**에 따라 적절한 mock 패턴을 선택하는 가이드.

## 품질 진단 기준

기존 mock 코드를 읽고 아래 항목으로 품질을 판단한다.

| 항목 | 불량 | 양호 |
|------|------|------|
| 반환값 | 하드코딩된 단일 값 | 입력에 따라 다른 결과 |
| 데이터 건수 | 1~2건 | 5건 이상 |
| 에러 시뮬레이션 | 없음 | 특수 입력값으로 트리거 가능 |
| 출력 스키마 | 명세서와 불일치 | `<output_format>` 구조 정확히 일치 |
| 데이터 현실성 | `"test"`, `"foo"` 등 더미 | 도메인에 맞는 현실적 값 |

## 패턴 선택 기준

명세서의 Tool Definitions에서 각 도구의 Purpose와 Signature를 보고, 아래 기준으로 개선 패턴을 선택한다.

| 도구 성격 | 판별 기준 | 패턴 |
|----------|----------|------|
| **검색/탐색** | 자연어 쿼리 → 유사도 기반 결과 목록 반환 | Keyword Matching |
| **코드/분류 조회** | 정확한 ID/코드 → 단건 또는 소수 결과 반환 | Lookup Table |
| **상태 변경** | 생성/수정/삭제 요청 → 성공/실패 응답 | Stateful CRUD |
| **계산/변환** | 수치 입력 → 계산된 수치 출력 | Deterministic Compute |
| **외부 시스템 연동** | 알림 전송, 웹훅, 이메일 등 사이드이펙트 | Fire-and-Confirm |

복합 도구(검색 + 계산 등)는 패턴을 조합한다.
